## Performance Audit

### Spreadsheet service bottlenecks
| Impact | Function / pattern | Issue | Recommended optimization |
|---|---|---|---|
| Critical | `readDashboardSectionRows_` | Uses `getRange(...).getValue()` and `getRange(...).getValues()` inside a row loop while scanning dashboard sections. | Read the entire section block once with `getDisplayValues()`/`getValues()`, then process in memory. |
| High | `enforceDashboardSectionDefaults_` | Calls `setValue()` inside a loop for each dashboard default correction. | Build a modified values array and write the affected dashboard range once. |
| High | `copyDemoPToMasterList_` | Inserts columns inside a loop and can call `getMaxColumns()` repeatedly. | Calculate missing columns once; insert columns in a single operation. |
| High | `getFirstValidDateFromCells_` | Loops through candidate cells and performs per-cell `getRange`, `getValue`, and `getDisplayValue`. | Fetch candidate ranges as a RangeList if possible, or pass already-read header/date values from caller. |
| Medium | Title/header setup paths | Many adjacent `setValue()` calls (`A1`, `A2`, `B2`, `C2`, `D2`) repeat across sheets. | Use one `setValues()` call for `A1:D2` title/date blocks and reuse helper. |
| Medium | `activate()` use | Several workflow helpers activate sheets. | Avoid activation except final UX step; use direct sheet references. |
| Medium | `SpreadsheetApp.flush()` | Flushes are limited to final date-format enforcement and intentional width-mismatch tests, but still expensive. | Keep only where validation immediately depends on committed formatting; remove incidental flushes. |

### Data processing bottlenecks
- Sorting blocks and row arrays is generally in-memory and acceptable, but repeated full-sheet scans should be cached per sheet version.
- Header maps are cached in several helpers; continue this pattern and invalidate only after writes that affect headers.
- Repeated `.filter()` comparisons over large header lists can be replaced with `Set` membership where called repeatedly.

## Google Apps Script Best Practices

- **Execution limits:** Full monthly report + master-list update + index refresh in one UI action is a timeout risk for large member populations.
- **Quota efficiency:** Most data tables use batch reads/writes, but remaining cell-in-loop hotspots should be eliminated first.
- **Trigger safety:** Prompting/UI workflows should not be used directly by time-driven triggers; maintain trigger-safe worker functions.
- **Exception handling:** The main workflow catches and rethrows major phase errors, which is good; add persistent error logging for non-interactive runs.
- **Large sheet handling:** Consider chunked writes for very large outputs and consistent maximum-row guards for quality dashboard scans.

## Refactoring Recommendations

Do not rewrite immediately; refactor in controlled phases:

1. **Create modules by responsibility:** `Config.gs`, `Dates.gs`, `SheetLookup.gs`, `MonthlyReport.gs`, `MasterListUpdate.gs`, `ChangeLog.gs`, `Dashboard.gs`, `Formatting.gs`, `QualityTests.gs`.
2. **Introduce a workflow context object:** carry spreadsheet, month parts, source sheets, cached header maps, and timing logger instead of repeated lookups.
3. **Deprecate no-op helpers:** mark with `@deprecated` for one release, remove after confirming no menu/manual use.
4. **Centralize title/date block writes:** one helper for standard `A1:D2` headers.
5. **Separate interactive and trigger-safe execution:** prompt functions should only collect parameters; worker functions should accept explicit dates.
6. **Build static dependency report tooling:** generate Called By / Calls To / orphan status as part of release review.

## Completed Recommendation Implementation Notes

The first implementation pass completed the highest-confidence, low-risk recommendations from this review:

- Batched Master List title/date writes from repeated single-cell `setValue()` calls into `A1:D2` `setValues()` writes.
- Replaced column-by-column Master List header-copy growth with one computed missing-column insertion.
- Converted Dashboard Sheet Definition Governance corrections from per-cell `setValue()` calls inside the governance loop to an in-memory update followed by one `setValues()` write.
- Converted Dashboard section row scanning to use the already-read sheet values array instead of additional `getRange().getValue()` and row `getValues()` calls inside the scan loop.

Deferred items remain architectural or behavior-changing recommendations and should be handled in separate releases: trigger-safe workflow separation, module extraction, no-op helper deprecation/removal, persistent trigger logging, and full static dependency tooling.
